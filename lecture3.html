Week 3: Translation

This week, we will learn about translating between the language of symbolic logic and natural language. There won't be a lot of theory in this lecture: it will be rather hands-on and practical.

From English Into Logic

Up until now, we've been working entirely within the language of symbolic logic. You've had the chance to write down sentences in symbolic notation, and then use the World app to see if those sentences are true in the provided world. The limitation is that this only works for sentences that describe our little world: sentences that talk about squares, diamonds and octagons and their relation to each other on the board. But there's a lot more to life than squares, diamonds and octagons, and symbolic logic can be used to describe any type of subject matter you want. This week, we're going to explore this aspect of logic by showing you how to translate English sentences into logical notation.

So suppose I give you a sentence like "Michael is happy." How would you go about translating that sentence into logical notation? The first thing you have to do is decide what names and predicates to use. In particular, you will need a name that refers to Michael, and a predicate that means happy. In traditional logic texts, names and predicates are usually limited to one letter each, so we would have to choose, say, m for Michael and H for happy. But there was never a very strong rationale for this requirement, and in this course you can choose names and predicates that are easier to remember. Let's go ahead and choose a name and a predicate:

  • michael will be our name for Michael; and
  • IsHappy will be our predicate meaning happy.

With that out of the way, we can now write down our sentence, "Michael is happy," in symbolic notation:

  • IsHappy(michael)

We can go on to say other things in this notation, such as: Michael is not happy:

  • ~IsHappy(michael)

Now suppose we create a new name sarah and a new predicate IsSad. Then we can say lots of new things in logical notation. Of course, we can say that Sarah is sad:

  • IsSad(sarah)
But we can also say that Michael is sad, or that Sarah is happy, or that Michael is sad and Sarah is happy:
  • IsSad(michael)
  • IsHappy(sarah)
  • IsSad(michael) & IsHappy(sarah)

We can say that if Michael is sad, then Sarah is happy:

  • IsSad(michael) -> IsHappy(sarah)

We can say that if Michael is happy, then Sarah is either happy or sad:

  • IsHappy(michael) -> (IsHappy(sarah) | IsSad(sarah))

And so on. Hopefully you get the general idea.

More Software Goodness

Before getting further into translation, let's look at a new software tool, the translation checker. Let's start with an example. Consider the English sentence "Michael is happy and Sarah is sad." By now you should know how to translate that sentence into logical notation. Go ahead and enter what you think is the correct translation into the text box below, and click the check button:

There are three responses you can get: "Correct," "Incorrect," and "Ill-formed." An ill-formed sentence is one that doesn't follow the rules of the language, and if you get that as a response, it simply means you made a mistake entering your sentence, such as a stray symbol. In any case, please feel free to enter different sentences above to see what you get.

Before going further, it's worth explaining how the translation checker works. In short, it doesn't work by magic, so don't expect it to. Each translation checker box comes pre-programmed with a sentence in logical notation, which is known to be a correct translation of the corresponding English sentence. The checker parses the sentence you entered, and determines whether it is logically equivalent to the expected translation. If it is, your answer is considered correct; otherwise it isn't.

Because of how the translation checker works, it has some limitations. First of all, you have to use the expected names and predicates. For example, suppose that instead of entering IsHappy(michael), you entered Happy(michael). That sentence wouldn't be accepted, because the translation checker is expecting you to use the predicate IsHappy, not Happy. Likewise, the translation checker won't accept IsHappy(mike) if the expected name is michael. Misspellings can cause your sentences to be marked incorrect for the same reason. Thus, when using the translation checker, you may also need a spell checker.

On the other hand, the translation checker actually analyzes the sentences you enter; it doesn't just compare them letter-by-letter to the expected response. This gives you some flexibility. For example, as you surely know by now, the correct answer to the translation exercise I gave you earlier is IsHappy(michael) & IsHappy(sarah). However, that sentence is logically equivalent to the sentence IsHappy(sarah) & IsHappy(michael), and so the latter would be accepted as well.

Incidentally, the notion of logical equivalence is an important concept in its own right, which we will be studying in a few weeks. For now, though, you don't need to know any particular facts about logical equivalence. Just think of logically equivalent sentences as those that "say the same thing."

Tips and Techniques

By far the best approach to translating a sentence is to (1) think about what the sentence means, and then (2) think about how to express that meaning in logical notation. This requires you to actually think in logical notation to some extent. The same thing goes, by the way, for learning a natural language: you need to learn to think in the new language.

Unfortunately, this is not the approach that all beginning logic students take, just as not all language learners really learn to think in the target language. And this can lead to some serious mistakes. In the case of learning a foreign language, the problem is that the learner will impose the grammar and idioms of her native language on the foreign language, producing a result that is incorrect in the foreign language. (In one real life example of my acquaintance, a native speaker of Spanish said: "I sometimes get emails that don't like me." He meant to say "I sometimes get emails that I don't like"; but in the Spanish construction for "I like," the roles of subject and object are reversed relative to English, and he failed to make the necessary correction. Of course, native speakers of English make equally bad mistakes when they learn Spanish.)

The same thing can happen when translating into logical notation: a sentence will sometimes take a different form in logical notation than it does in English. To give a simple example, suppose we want to translate "Both Amy and Brett like cookies." If you are too focused on how this sentence is constructed in English, you might try to render it as follows in logical notation: LikesCookies(amy & brett). But this would be WRONG. In logical notation, the connective & can only join two sentences: it can never join two names. So amy & brett is just ill-formed. By contrast, English allows two names to be joined with and, so "Amy and Brett" is just fine.

Breaking Things Down

One good technique for translation is to break down a sentence bit by bit. Here's an example of what I mean. Suppose we start with a rather long and complicated English sentence, such as the following:

If Michael and Sarah both have cats, then either Michael is happy, or Michael is sad and Sarah and Amy are both happy.

As a first step, we notice that the sentence is an if-then sentence, and so it is going to have to take the form A -> B. With this in mind, we write down the following:

(Michael and Sarah both have cats) -> (either Michael is happy, or Michael is sad and Sarah and Amy are both happy)

(I'm showing the translated parts in red.) We can now work on the parts before and after the arrow separately, which greatly simplifies the task. Let's start with the part before the arrow, which is called the antecedent.

(Michael has cats & Sarah has cats) -> (either Michael is happy, or Michael is sad and Sarah and Amy are both happy)

Here we've broken the antecedent down one step further. We've decided that the antecedent will take the form A & B, and we just have to decide what A and B will be. Fortunately, that's not too difficult.

(HasCats(michael) & HasCats(sarah)) -> (either Michael is happy, or Michael is sad and Sarah and Amy are both happy)

We have now fully translated the antecedent, and we can get to work on the part after the arrow, which is called the consequent. Notice now that the consequent is an either-or statement: it says that either Michael is happy, or Michael is sad and Sarah and Amy are happy. We know that either-or statements take the form A | B, so we can proceed as follows:

(HasCats(michael) & HasCats(sarah)) -> ((Michael is happy) | (Michael is sad and Sarah and Amy are both happy))

And of course we know how to say "Michael is happy" in logical notation, so we can substitute that in as follows:

(HasCats(michael) & HasCats(sarah)) -> (IsHappy(michael) | (Michael is sad and Sarah and Amy are both happy))

Now we just have to translate "Michael is sad and Sarah and Amy are both happy," which is a much simpler task than what we started with. Again, we can break that sentence down into simpler ones:

(HasCats(michael) & HasCats(sarah)) -> (IsHappy(michael) | (IsSad(michael) & (Sarah and Amy are both happy)))

Finally, we just have to translate "Sarah and Amy are both happy," and we're done:

(HasCats(michael) & HasCats(sarah)) -> (IsHappy(michael) | (IsSad(michael) & (IsHappy(sarah) & IsHappy(amy))))

You might notice that the above sentence has more parentheses than are strictly needed. We can remove one pair if we like, which gives us the following sentence:

(HasCats(michael) & HasCats(sarah)) -> (IsHappy(michael) | (IsSad(michael) & IsHappy(sarah) & IsHappy(amy)))

Conditionals

One area where you have to be particularly careful is with "if" statements, which are called conditionals.

A conditional takes the form A -> B. The sentence A — the part before the arrow — is called the antecedent. The sentence B — the part after the arrow — is called the consequent. Please note that the word is consequent, with a t, not consequence with a c.

When you translate conditionals, you need to identify the antecedent and consequent of the English sentence, so that you know where to put their translations in the finished product. Sometimes this is easy. Let's say our sentence to translate is "If Brett has cats, then he is happy." Pretty clearly, this is translated as follows:

  • HasCats(brett) -> IsHappy(brett)

But now consider the sentence "Brett is happy if he has cats." How should that be translated? Well, that sentence has the same meaning as the sentence "If Brett has cats, then he is happy," so both should be translated the same way. Thus, the sentence "Brett is happy if he has cats" is translated: HasCats(brett) -> IsHappy(brett).

What's tricky here is that in the sentence "Brett is happy if he has cats," the antecedent actually comes second, and the consequent comes first, which is the reverse of the usual order. Namely, "Brett is happy" is the consequent, even though it comes first, and as the consequent, it will have to come after the arrow in our translation. Likewise, "he has cats" is the antecedent, even though it comes second, and so it will have to come before the arrow in our translation. A simple rule of thumb is as follows: What follows 'if' is always the antecedent, regardless of where it occurs in the sentence.

Next, consider the sentence "Brett is happy only if he has cats." How do we translate that? While it may not be obvious, this sentence has the same meaning as "If Brett is happy, then he has cats." (If you're having trouble seeing this, consider that "Brett is happy only if he has cats" is equivalent to: "If Brett doesn't have cats, then he's not happy." The latter sentence, in turn, is logically equivalent to "If Brett is happy, then he has cats.") Thus, the sentence is translated IsHappy(brett) -> HasCats(brett).

Next, consider "Only if Brett has cats is he happy." That sentence has the same meaning as "Brett is happy only if he has cats," so both sentences have to translated the same way. Thus, the sentence is translated IsHappy(brett) -> HasCats(brett). Here, we have another simple rule of thumb: What follows 'only if' is always the consequent, regardless of where it appears in the sentence. The following chart summarizes the situation:

The Sentence...Is Translated...
If A then B A -> B
A if B B -> A
A only if B A -> B
Only if A, B B -> A

The above table is equivalent to the following pair of rules:

  • The word 'if' is always followed by the antecedent of a conditional.
  • The phrase 'only if' is always followed by the consequent of a conditional.

Unless

Translating sentences that contain 'unless' is a bit tricky. What you have to remember is that 'unless' means 'if not.' That is, 'A unless B' means 'A if not B'. Using the above chart, we see that this is translated ~B -> A. So, 'Sarah is happy unless Michael is sad' has the same meaning as 'If Michael is not sad, then Sarah is happy,' and therefore translates as follows: ~IsSad(michael) -> IsHappy(sarah).

Now what if we see a sentence like this: 'Unless Michael is sad, Sarah is happy'? Again, we use the rule that unless = if not. Our sentence then means 'If Michael is not sad, then Sarah is happy.' The translation is therefore: ~IsSad(michael) -> IsHappy(sarah).

As we'll see in a couple of weeks, the sentence ~A -> B is logically equivalent to A | B. The sentence A | B in turn is equivalent to B | A. So 'A unless B' can also be translated as A | B. We've just discovered that 'unless' means 'or'!

Here's our chart again, expanded to inclue 'unless.' Where two translations are given, either can be used.

The Sentence...Is Translated...
If A then B A -> B
A if B B -> A
A only if B A -> B
Only if A, B B -> A
A unless B ~B -> A, A | B
Unless A, B ~A -> B, A | B
